# Git Commit Message

## 工作流程

1. 读取 [提交信息规范](references/commit-message-rules.md) 并严格执行。
2. 运行 `git status --short` 确认变更范围。
3. 优先分析暂存区：运行 `git diff --cached --stat` 和 `git diff --cached`。
4. 暂存区为空时，分析工作区：运行 `git diff --stat` 和 `git diff`，并查看 `git status --short` 列出的未跟踪文件。
5. 根据变更的主要意图选择一个提交类型，而不是根据文件扩展名机械判断。
6. 生成一条可以直接填入 Git 提交框的标题。

## 输出要求

- 默认只输出一行提交信息，不使用 Markdown、引号、代码块、说明或候选列表。
- 默认使用中文描述；仓库已有明确的其他语言规范时，沿用仓库规范。
- 不自动执行 `git add`、`git commit`、`git push` 或修改文件。
- 没有可分析的变更时，明确回复“当前没有可生成提交信息的 Git 变更”。
- 变更包含多个互不相关的主要意图时，不强行合并标题；简短指出应拆分提交，并分别给出建议标题。
- 用户明确要求解释、多个候选或执行提交时，再按用户要求扩展输出或执行操作。

## 判断原则

- 以“为什么修改”为主，以“修改了什么”为辅。
- 修复错误行为使用 `fix`，即使改动主要发生在 CSS 或配置文件中。
- 只调整视觉呈现、格式或代码样式且不修复错误行为时使用 `style`。
- 新增用户可感知能力使用 `feat`；内部整理且行为不变使用 `refactor`。
- 标题只描述本次变更中最重要、可验证的结果，不罗列文件名和实现步骤。
