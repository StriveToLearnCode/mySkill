# 提交信息规范

## 格式

使用单行格式：

```text
<type>: <description>
```

默认不添加 scope、正文、句号或结尾标点。描述应具体、简洁，建议控制在 50 个中文字符以内。

## 类型选择

| 类型 | 使用场景 |
| --- | --- |
| `feat` | 新增功能、能力或用户可感知行为 |
| `fix` | 修复 Bug、错误行为、异常状态或回归问题 |
| `style` | 调整 UI 视觉、CSS、排版、格式化或代码样式，且不改变业务行为 |
| `refactor` | 重构内部实现，外部行为保持不变 |
| `perf` | 提升性能、速度、内存或资源使用效率 |
| `docs` | 只修改文档、注释或使用说明 |
| `test` | 新增或调整测试，不改变业务实现 |
| `build` | 修改构建系统、依赖或打包配置 |
| `ci` | 修改 CI/CD 工作流或自动化配置 |
| `chore` | 其他维护性工作，无法归入以上类型 |
| `revert` | 撤销已有提交或变更 |

## 冲突判断

- 修复页面错位、点击失效或错误颜色等已有缺陷：使用 `fix`。
- 主动调整主题、间距、字号或动画效果，没有对应缺陷：使用 `style`。
- 新增按钮及其完整交互：使用 `feat`，不要因为包含 CSS 就使用 `style`。
- 升级依赖以修复明确 Bug：优先使用 `fix`；常规依赖升级使用 `build`。
- 同时修改实现和测试：按实现目的选择 `feat`、`fix` 或 `refactor`，不要使用 `test`。
- 只有测试文件发生变化：使用 `test`。

## 描述规则

- 使用“添加、修复、优化、调整、移除、更新、重构”等明确动词。
- 描述结果和对象，例如“修复登录按钮重复提交问题”。
- 不写“修改代码”“更新文件”“一些调整”“处理问题”等空泛内容。
- 不罗列文件名、函数名或实现步骤，除非它们是理解提交目的所必需的领域名称。
- 不使用 `and`、斜杠或顿号拼接多个独立意图；这种情况应拆分提交。

## 示例

```text
fix: 修复登录按钮重复提交问题
style: 调整个人中心页面间距和字体样式
feat: 添加订单导出功能
refactor: 简化用户权限校验逻辑
docs: 补充本地开发环境配置说明
test: 添加购物车金额计算测试
build: 升级 Vue 和 Vite 依赖
ci: 添加主分支构建检查
chore: 清理未使用的开发脚本
```
