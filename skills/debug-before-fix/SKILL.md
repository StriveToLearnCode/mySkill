# Debug Before Fix

## 核心原则

先用证据定位，再提出最小修复方案。未经用户确认，不修改业务逻辑。

固定流程：

1. 找出最短失败链路。
2. 查看相关代码、最近变更、输入参数、网络请求、状态变化和报错信息。
3. 使用影响最小的调试方式定位问题。
4. 找到第一处有证据支持的异常后停止继续猜测。
5. 告诉用户问题位置、证据、根因和最小修复方案。
6. 等待用户明确确认。
7. 只实施用户确认的最小修复。
8. 验证修复结果。
9. 只读扫描是否存在同类 Bug，并逐项告诉用户。
10. 同类 Bug 得到单独确认后才能修改。
11. 验证完成后清理临时日志、调试代码和断点。

## 调试方式

根据运行环境选择合适的方式：

- Web Bug：优先使用浏览器 DevTools、Source 断点、条件断点、Network、Vue DevTools、React DevTools 和现有日志。
- 真机 Bug：使用远程调试和统一前缀日志。
- 异步 Bug：检查函数调用顺序、Promise、callback、nextTick、定时器、组件挂载和状态更新时间。
- 接口 Bug：检查请求参数、响应关键字段、调用次数、前后状态和错误分支。
- 其他运行环境：使用调试器断点、监视表达式、trace 或针对性测试。

优先使用不修改源码的断点。如果必须向源码加入临时日志，而用户没有提前授权调试代码修改，先说明要加在哪里、打印什么，并等待确认。

## 必须检查的关键节点

根据链路打印或观察：

- 函数是否进入。
- 输入参数是否正确。
- 接口是否调用及返回了什么。
- 分支条件结果。
- 状态修改前后的值。
- 列表变化前后的数量和稳定 ID。
- callback 是否执行。
- nextTick 后 ref 是否存在。
- 动画是否开始、结束。
- 弹窗打开前的数据是否完整。
- 第一个与预期不一致的节点。

## 证据规则

- 明确区分“事实”“推测”和“未知”。
- 没有运行日志、断点值、可复现测试、明确代码路径或状态矛盾时，不能直接声称已经找到根因。
- 暂时无法复现时，使用“当前推测”，并向用户索取最少的必要证据。
- 对比第一处发生分歧的预期值和实际值。
- 异步链路必须验证执行顺序，不能根据代码位置猜测执行顺序。
- 对比列表时，同时检查数量和稳定 ID，不能只看最终数组。
- 如果累计值出现 `afterCount < beforeCount`，优先检查数据是否按阶段、轮次、Tab、账号或活动重置。

## 统一日志规范

日志使用简短且唯一的功能前缀，例如 `[lottery]`、`[payment]`、`[modal]`。

```ts
console.log("[lottery] showRewards:start", { input });

console.log("[lottery] progress:diff", {
  beforeCount,
  afterCount,
  beforeIds,
  afterIds,
  giftRewards,
});

console.log("[lottery] callback:called");

console.log("[lottery] nextTick:ref", {
  exists: Boolean(rouletteRef.value),
});

console.warn("[lottery] result:not-found", {
  result,
  candidates,
});
```

只记录能够区分链路分支的必要数据。禁止输出 Token、个人信息、支付凭证和其他敏感信息。

## 向用户索取日志的话术

> 先不要改业务逻辑。我会通过断点或统一前缀日志定位问题。请完整复现一次失败流程，在控制台或手机日志中搜索 `[功能前缀]`，把从 `start` 到最后一条同前缀日志完整贴回来，不要只截最后一条。我会根据真实执行链路判断第一处异常，再给出最小修复方案。

## 修改前必须输出的报告

```markdown
定位结果：

- 问题位置：文件 / 函数 / 链路节点
- 证据：日志、断点值、请求、测试或代码路径
- 根因：已证实的第一处异常；没有完全证实时写“待确认”

最小修复方案：

- 准备改什么
- 为什么这样改
- 不会改什么
- 可能风险
- 验证方式

同类问题扫描：

- 已发现：位置、证据、是否属于同一根因、建议方案
- 未发现：说明已经扫描的范围

等待确认：

- 请确认“只修当前问题”或“当前问题 + 指定同类问题”
```

不要把多个未经验证的猜测都包装成可直接执行的方案。优先推荐证据最充分、改动最小的方案，并说明真实取舍。

## 最小修复规则

- 只修复恢复预期行为所必需的第一处已证实问题。
- 不修改无关 UI。
- 不修改接口协议。
- 不重构整条业务链路。
- 没有证据和用户确认，不修改匹配字段。
- 不进行无关命名和代码清理。
- 尽量增加一个针对当前问题的回归测试，但不扩展成大范围测试重构。
- 保持失败链路之外的现有行为不变。
- 同类问题在获得确认之前只能只读扫描，不能顺手修复。

## 同类 Bug 排查规则

当前问题的根因得到证实后，搜索附近代码、相同组件、公共方法和相同调用模式。

每个同类问题分别说明：

- 文件和函数位置。
- 与当前 Bug 相同的代码模式。
- 是否有运行证据，还是静态风险。
- 影响范围。
- 建议的最小修复。
- 是否建议本次一起处理。

不能因为用户批准了当前 Bug 的修复，就默认获得了修改所有同类问题的权限。

## 修复完成后的报告

修复后必须告诉用户：

- 修改了哪个文件和函数。
- 具体修改了什么。
- 哪些内容明确没有修改。
- 做了什么验证，结果如何。
- 用户如何重复验证。
- 临时日志和调试代码是否已清理。
- 是否仍有等待用户确认的同类问题。

如果验证结果推翻了原诊断，应在安全的情况下只撤回本次尝试的调试修复，告诉用户新的证据，然后重新回到定位阶段，不能继续叠加猜测式修改。
